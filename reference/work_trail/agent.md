# 一、项目背景

我们是一家定制化软件系统开发公司，团队规模约 20 人左右，有 2-3 位主要产品经理负责需求拆分和任务分发。业务同时包含以下几类：

## 1. 大型项目
- 项目周期长
- 有明确项目管理诉求
- 任务层级可能较深
- 需要按项目、模块、阶段、成员统计投入

## 2. 小型敏捷项目
- 节奏快
- 任务较碎
- 适合迭代、Sprint、看板式管理
- 需要快速分发和执行反馈

## 3. 移动端 App 孵化 / 探索类工作
- 存在较多预研、验证、讨论、尝试性工作
- 并非所有工作都天然对应一个正式任务
- 仍需要记录投入与产出归因

历史上已有基础工时填报系统，但体验较差，只能填写：
- 今天工作了多长时间
- 属于哪个项目
- 写一些备注

现在希望设计并持续迭代一套新的系统，核心目标不是做一个更漂亮的表单，而是做一个：

**“以任务/工作项为核心、以时间块交互为入口、以过程分析和管理分析为输出”的系统。**

# 二、系统定位

请始终把该系统理解为：

**“面向软件研发团队的任务驱动式可视化工时与执行分析系统”**

它不是：
- 纯考勤系统
- 纯项目管理系统
- 纯任务管理系统
- 重型审批流系统
- Jira/TAPD/禅道的完整替代品

它的核心职责是：
- 记录工作时间块
- 关联项目 / 工作项 / 工作类型
- 还原执行过程
- 为管理分析提供结构化数据
- 在不增加员工负担的前提下提升可分析性

# 四、关键设计原则

你在输出任何方案时，必须遵循以下原则：

## 1. 先体验，后管理
先确保员工愿意填、填得快，再考虑分析与管理。

## 2. 任务驱动优先，但允许自由录入
不是所有工作都来自任务分发，因此系统必须同时支持：
- 按工作项录入
- 按项目录入
- 按非任务型工作录入

## 3. 日历 / 时间轴优先
主交互优先采用以下形式：
- 日视图
- 周视图
- 时间轴视图
- 时间块拖拽录入
- 时间块拖动、拉伸、复制

## 4. 结构化优先于备注
返工率、完成度、延期、阻塞等分析，不能依赖自由文本备注，必须优先通过结构化字段、状态流转、事件记录来实现。

## 5. 轻量优先于完整
当存在“完整但更重”和“稍轻但够用”两种方案时，优先选择轻量方案。

## 6. 任务完成必须显式触发
拖动工作项到日历上，不等于完成。完成必须是独立的状态变更动作。

## 7. 管理同步要轻，不要把所有状态变化都变成待办
Done 更偏通知型事件，Blocked / Reopened / Overdue 更偏待处理型事件。

## 8. 信息密度优先于装饰感
对于高频操作型页面，应优先使用：
- 紧凑列表
- 表格
- 分组列表
- 时间轴
- 迷你进度条
- 风险列表
而不是大量低信息密度的大卡片

# 六、核心模型要求

## 1. Project
Project 用于归属、过滤和分析，不要一开始做成重型计划管理对象。

建议包含：
- 项目名称
- 项目编码
- 项目类型（定制项目 / 敏捷项目 / 孵化项目 / 内部项目）
- 项目状态（进行中 / 暂停 / 已结束）
- 默认成员
- 负责人 / PM
- 可选字段：客户、日期范围、是否可计费

## 2. Work Item
Work Item 是比 Task 更宽的概念，用来兼容正式任务和非正式工作项。

Work Item 类型建议支持：
- Task
- Bug
- Requirement
- Meeting
- Research
- Support
- Admin

Work Item 建议包含：
- 标题
- 所属项目
- 类型
- 负责人
- 分发人
- 状态
- 截止时间（可选）
- 预估工时（可选）
- 实际累计工时
- 优先级（可选）
- 标签（可选）

## 3. Time Block
Time Block 是日历上的时间块，是系统最核心的对象。

每条 Time Block 至少应关联以下之一：
- 项目
- 工作项
- 非任务型工作项

建议字段：
- 日期
- 开始时间
- 结束时间
- 时长
- 所属员工
- 所属项目
- 关联工作项（可选）
- 工作类型
- 工作内容摘要
- 是否返工
- 返工原因
- 是否阻塞
- 阻塞原因
- 是否加班
- 创建来源（拖拽创建 / 从工作项生成 / 手动补录 / 批量复制）

# 七、任务状态与完成逻辑

## 1. 工作项状态必须明确，不使用单纯 checkbox 布尔值
推荐状态：
- Todo
- In Progress
- Blocked
- Done
- Reopened

## 2. 拖入日历不等于完成
规则如下：
- 拖入日历 = 创建时间块
- 如果该工作项原状态是 Todo，第一次拖入后可自动变为 In Progress
- 再次拖入，只表示继续投入时间
- 不要因为拖入日历而自动变成 Done

## 3. 完成必须是独立动作
完成应通过以下方式之一触发：
- 工作项列表 / 卡片中的快捷状态操作
- 时间块详情中的“保存并标记完成”
- 工作项详情中的状态切换

## 4. 已完成后再次记录时间
如果 Done 状态的工作项再次被拖入日历，系统应提示用户：
- 仅补充历史工时
- 重新打开工作项

## 5. Blocked 和 Reopened 必须支持
Blocked 用于表达当前需要外部协助或信息澄清。  
Reopened 用于表达已完成工作再次进入执行。

# 八、主管 / PM 可见性与提醒逻辑

不要把所有状态变化都做成强制审批或强制待办。

## 1. Done
- 对主管 / PM 可见
- 进入 “Recently Completed / 最近完成” 或 “Pending Review / 待查看”
- 默认不是强制待办，除非明确启用了验收模式

## 2. Blocked
- 对主管 / PM 可见
- 进入 “Needs Attention / 需关注”
- 应视为更接近待处理事项

## 3. Reopened
- 对主管 / PM 可见
- 进入 “Needs Attention” 或 “Rework Risk / 返工风险”

## 4. Overdue
- 自动进入风险列表
- 作为项目管理风险信号

## 5. 产品要求
主管看到的是“关注事件队列”，不是把所有状态变化都堆成待办中心。

# 九、返工率与完成度分析要求

## 1. 返工率不能靠备注推断
优先通过以下结构化方式识别：
- 工作项从 Done 回到 Reopened
- 工作项状态从完成态回退到执行态
- Time Block 被标记为 Rework / Bug Fix
- 返工原因被结构化记录

返工原因建议支持：
- 需求变更
- 理解偏差
- 设计缺陷
- 测试发现问题
- 客户反馈修改
- 依赖问题
- 质量不达标

## 2. 完成度不要主要依赖人工百分比
优先支持：
- 状态映射完成度
- 实际工时 vs 预估工时
- 子任务加权汇总完成度（可后置）
- 风险提示（投入过高但状态进展不足）

# 十、页面结构与低负担展示要求

## 1. 工作台页面
目标：
- 帮员工快速知道今天做什么
- 在当前页面就能开始记录
- 减少无效装饰性卡片

建议布局：
- 左侧：我的工作项（紧凑分组列表）
  - Assigned to Me
  - In Progress
  - Blocked
  - Due Soon
- 中间：日历 / 时间轴（主工作区）
- 右侧：提醒与摘要
  - Recently Completed
  - Missing Logs
  - Quick Actions
  - Recent Projects / Recent Work Items

可加一条顶部 summary strip：
- Today Hours
- In Progress Count
- Blocked Count
- Due Soon Count

## 2. 项目页面
不要做大卡片墙。建议：
- 顶部：紧凑 summary strip
- 下方 tab：
  - Work Items
  - Time
  - People
  - Risks

内容主体优先使用：
- 表格
- 分组列表
- 简单趋势图
- 风险清单

## 3. 工作项 / 任务页面
默认视图应为高密度列表 / 表格，而不是大卡片平铺。

字段建议：
- 标题
- 项目
- 类型
- 负责人
- 状态
- 优先级
- 截止时间
- 预估工时
- 实际工时
- 差值
- 最后活动时间

看板视图可以作为辅助视图，不应是唯一主视图。

## 4. 分析页面
不要做 KPI 大卡片墙。建议：
- 顶部最多 4 个关键指标
- 中部：趋势图 / 堆叠条图
- 下部：问题对象排行表

优先分析：
- 项目投入
- 工作类型分布
- Blocked 数量
- Reopened 数量
- Overdue 项
- Estimate vs Actual 差异

# 十四、Codex 执行规则

当你实际修改代码时，必须遵循以下规则：

## 1. 先读代码，再改代码
你必须先理解当前仓库的：
- 路由结构
- 页面结构
- 状态管理方式
- 日历交互实现方式
- 当前项目/任务模型

## 2. 保持主链路稳定
优先保证以下主链路不被破坏：
- 工作项 -> 时间块录入 -> 工时保存 -> 基础统计

## 3. 改动优先小步增量
每次优先做最小可运行改动，不要一次性重构整个系统。

## 4. 输出结果时要说明
每次实现功能时，说明：
- 改了哪些文件
- 为什么这样设计
- 还缺什么
- 哪些内容留到后续阶段

## 5. 页面改造优先级
当前页面改造优先级：
1. 工作台信息密度提升
2. 工作项状态模型完善
3. 明确完成 / 阻塞 / 重开逻辑
4. 项目页从卡片墙改为高密度 tab 结构
5. 分析页从 KPI 卡片墙改为图表 + 排行表

## 6. 避免这些问题
- 不要把拖入日历直接等同于完成
- 不要把所有状态变化都做成审批流
- 不要用大量大卡片替代高密度信息布局
- 不要让员工必须填写过多字段
- 不要引入与当前目标无关的大量样板代码
- 不要做重型 PM 系统
